TLS 指纹审核报告
一、Chrome 131 指纹审核
1. JA3/JA4 — 结构格式正确
项目	状态	说明
JA4 格式 t13d0911h2	正确	t=TCP, 13=TLS1.3, d=有SNI, 09=密码套件数, 11=扩展数, h2=ALPN
JA3 version=771	正确	TLS 1.2 legacy version (0x0303=771)
签名算法 (13个)	正确	与真实 Chrome 131 一致
2. 密码套件 — 缺少 legacy 套件 (高风险)
你的实现只有 9 个密码套件：


4865,4866,4867,49195,49199,49196,49200,52393,52392
真实 Chrome 131 还包含以下 legacy 套件 用于向后兼容：

缺少的套件	ID (hex)	ID (dec)	名称
缺少	0xC013	49171	TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA
缺少	0xC014	49172	TLS_ECDHE_RSA_WITH_AES_256_CBC_SHA
缺少	0x009C	156	TLS_RSA_WITH_AES_128_GCM_SHA256
缺少	0x009D	157	TLS_RSA_WITH_AES_256_GCM_SHA384
缺少	0x002F	47	TLS_RSA_WITH_AES_128_CBC_SHA
缺少	0x0035	53	TLS_RSA_WITH_AES_256_CBC_SHA
这直接导致 JA4 中的密码套件数量是 09 而非真实 Chrome 的 15，非常容易被检测。

3. ClientHello Session ID — 缺少 32 字节随机 ID (中风险)
代码 (tls_fingerprint_generator.cc:123)：


body.push_back(0x00);  // Session ID (empty for TLS 1.3)
真实 Chrome 131 在 TLS 1.3 中仍然发送一个 32 字节的随机 session ID（用于 middlebox compatibility mode, RFC 8446 Appendix D.4）。空 session ID 是明显的非浏览器特征。

4. EC Point Formats — 只有 1 个格式 (低风险)
代码只发送 0(uncompressed)，真实 Chrome 发送 0, 1, 2（uncompressed + 两种压缩格式）。

5. 缺少 padding 扩展 (Extension 21) — (中风险)
Chrome 使用 padding 扩展将 ClientHello 填充到一致大小（通常 512 字节），generator 中没有实现。

6. GREASE 只加在 extensions 上 — (中风险)
真实 Chrome 在以下三个位置都插入 GREASE：

Extensions (已实现)
Cipher suites 列表头部 (未实现)
Named groups / Supported versions (未实现)
7. HTTP/2 Akamai 指纹 — 完美匹配

1:65536;2:0;3:1000;4:6291456;5:16384;6:262144;8:1|15663105|0|m,a,s,p
SETTINGS 参数、WINDOW_UPDATE 增量 (15663105)、pseudo-header 顺序 (m,a,s,p) 全部与真实 Chrome 一致。

8. Named Groups — 基本正确
x25519_mlkem768(4588) 是 Chrome 131 引入的 post-quantum 密钥交换，codepoint 正确。

二、Firefox 133 指纹审核
1. 密码套件 — 缺少 legacy 套件 (高风险)
你的实现只有 9 个套件，真实 Firefox 133 还包括：

缺少	ID	名称
缺少	49171 (0xC013)	TLS_ECDHE_RSA_WITH_AES_128_CBC_SHA
缺少	49172 (0xC014)	TLS_ECDHE_RSA_WITH_AES_256_CBC_SHA
缺少	0x009C (156)	TLS_RSA_WITH_AES_128_GCM_SHA256
缺少	0x009D (157)	TLS_RSA_WITH_AES_256_GCM_SHA384
缺少	0x002F (47)	TLS_RSA_WITH_AES_128_CBC_SHA
缺少	0x0035 (53)	TLS_RSA_WITH_AES_256_CBC_SHA
2. Named Groups — 问题: ffdhe 组 BoringSSL 不支持
配置中有 ffdhe2048(0x0100) 和 ffdhe3072(0x0101)，但 BoringSSL 不支持 FFDHe 组。SSL_set1_group_ids 调用时这些组会被静默忽略或导致错误，实际发送的 ClientHello 中可能没有这两个组。

对比检测到的 JA3 Text：


named_groups: 29-23-24
只有 3 个组（x25519, secp256r1, secp384r1），而真实 Firefox 应该有 6 个。缺少 secp521r1(25) 和 ffdhe 组说明了这个问题。

3. Firefox 不应有 GREASE 和 extension permutation — 正确
配置中 enable_grease = false, permute_extensions = false，这是对的。

4. Firefox 独有的扩展 — 缺少 delegated_credentials (中风险)
Firefox 支持 delegated_credentials (extension 34)，你的实现没有包含。

5. HTTP/2 指纹 — 基本正确

1:65536;3:100;4:131072;5:16384;6:262144|12517377|0|m,a,s,p
Firefox 特有的 SETTINGS 值（INITIAL_WINDOW_SIZE=131072 vs Chrome 的 6291456）已经正确区分。

三、Safari 18.2 指纹审核
1. 密码套件 — 基本正确但有疑问

4866,4867,4865,49195,49199,49196,49200 (7个)
Safari 将 AES-256 放在 AES-128 前面（4866 before 4865），这是对的。没有 CHACHA20 TLS 1.2 套件也是对的。

但真实 Safari 可能还包含一些 legacy 套件。

2. GREASE — Safari 应该有 GREASE (高风险)
研究确认 Safari 18.2 使用 GREASE (RFC 8701)。但你的配置中：


config.enable_grease = false;  // 错误！Safari 使用 GREASE
这是严重问题，会导致指纹与真实 Safari 不一致。

3. Named Groups — ffdhe2048 同样有 BoringSSL 不支持的问题
Safari 配置包含 ffdhe2048(0x0100)，但 BoringSSL 不支持。

四、Edge 131 指纹审核
Edge = Chrome (都是 Chromium)，所有 Chrome 的问题同样适用。JA4 和 JA3 完全相同是合理的，仅 User-Agent 不同。
